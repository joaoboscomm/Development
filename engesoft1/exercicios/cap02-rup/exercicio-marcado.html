O RUP é um exemplo de modelo de processo moderno derivado do trabalho sobre a UML e do PU (Processo Unificado de Desenvolvimento de Software). Traz elementos de todos os modelos genéricos de processo, apóia a iteração e ilustra boas práticas de especificação e projeto. O RUP reconhece que os modelos convencionais de processo apresentam uma visão única de processo. Por outro lado, o RUP é geralmente descrito a partir de três perspectivas: 1.Uma perspectiva dinâmica, que mostra fases do modelo ao longo do tempo 2.Uma perspectiva estática, que mostra as atividades realizadas no processo 3.Uma perspectiva prática, que sugere as boas práticas a serem usadas durante o processo. O RUP é um modelo construído por quatro fases. Ao contrário do modelo em cascata, no qual as fases coincidem com as atividades do processo, as fases do RUP estão relacionadas mais estritamente aos negócios do que a assuntos técnicos. As fases do RUP são: 1. Concepção : o objetivo é estabelecer um business case para o sistema. Identificar as entidades externas (pessoas e sistemas) que vão interagir com o sistema e definir essas interações. Após isso deve-se usar estas informações para avaliar a contribuição do sistema com o negócio. 2. Elaboração : os objetivos da fase são: a.Desenvolver um entendimento do domínio do problema b.Estabelecer um framework de arquitetura para o sistema c.Desenvolver o plano de projeto d.Identificar os riscos principais do projeto Ao término da fase tem-se: a.Um modelo de requisitos para o sistema (casos de uso da UML são especificados) b.Descrição de arquitetura c.Plano de desenvolvimento para o software 3. Construção : fase relacionada ao projeto, programação e teste de sistema. As partes são desenvolvidas paralelamente e integradas durante esta fase. Ao término da fase tem-se: a.Sistema em funcionamento b.Documentação associada pronta para ser liberada para o usuário. 4. Transição : transferência do sistema para os usuários com a entrada em funcionamento no ambiente real. A visão estática do RUP enfoca as atividades que ocorrem durante o processo de desenvolvimento. São chamadas de workflows. Existem 6 (seis) workflows principais identificados e 3 (três) workflows de apoio principais. A descrição dos workflows é orientada em termos dos modelos UML associados.
